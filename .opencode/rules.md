# Правила проекта презентации

## ⚠️ ОБЯЗАТЕЛЬНЫЙ WORKFLOW

**Любое изменение в проекте выполняется строго по шагам:**

### Шаг 1: План (ОБЯЗАТЕЛЬНО)
```
Перед любыми изменениями:
1. Использовать skill plan для создания плана
2. План сохранять в .opencode/plans/<название>.md
3. Показать план пользователю
4. Получить подтверждение на выполнение
```

### Шаг 2: Реализация
```
После подтверждения плана:
1. Использовать skill implement для выполнения
2. Выполнять шаги по порядку из плана
3. Обновлять статус в плане: [ ] → [x]
```

### Шаг 3: Контент (если нужен текст)
```
При написании контента:
1. Использовать skill content
2. Соблюдать стиль презентаций
```

### Шаг 4: Проверка
```
Перед коммитом:
1. Использовать skill review
2. Проверить HTML валидность
3. Проверить ссылки
4. Проверить обновлённый markdown
```

### Шаг 5: Публикация
```
После проверки:
1. git add . && git commit -m "описание"
2. git push origin main
3. Использовать skill deploy при проблемах
```

## Автоматический выбор skill

| Тип задачи | Skill |
|------------|-------|
| Новая фича / изменение | plan → implement → review |
| Написание текста | content → review |
| Исправление бага | implement → review |
| Публикация | deploy |

## Структура проекта
- `presentations/` — HTML-презентации (reveal.js)
- `content/` — исходные markdown-конспекты
- `reading/` — HTML-версии для чтения
- `assets/` — изображения и медиа

## Стиль презентаций
- Один слайд — одна идея
- Минимум текста, максимум смысла
- Использовать карточки для структуры
- fragment для пошагового показа

## Запреты
- ❌ НЕ пропускать шаг планирования
- ❌ НЕ коммитить без review
- ❌ НЕ использовать внешние скрипты без HTTPS
- ❌ НЕ забывать обновлять markdown при изменении HTML

## Git
- Сообщения коммитов на русском
- Один коммит — одно логическое изменение
- Пушить только после успешной проверки
